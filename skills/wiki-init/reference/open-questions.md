# `wiki-init` — Open Questions

> Supplementary material for `wiki-init`. Not loaded at LLM runtime — these are deferred design decisions for future iterations.

---

## 1. `.gitignore` handling

**Status:** MVP appends `deep/knowledge-graph.json` to `.gitignore` (or creates the file if absent), conditional on a config flag.

**Open:** what other entries should be auto-added? Candidates:
- `index/signatures.json` (regenerated by `wiki-sync`; arguably committable for diffability)
- `index/snapshots/sync_*.md` (immutable; researcher may want to commit history)

**Proposed:** stay minimal — only `deep/knowledge-graph.json` (which can be large). Researchers can extend `.gitignore` themselves. Decision documented in the initial config: `init.gitignore_entries: [deep/knowledge-graph.json]`.

---

## 2. Understand-Anything detection

**Status:** wiki-init does NOT auto-install Understand-Anything. It detects whether the binary is on `$PATH` and instructs the researcher to install if missing.

**Open:** should wiki-init at least produce a more actionable install instruction (e.g., the exact pip / brew command for the detected platform)?

**Proposed:** yes for v1.1. Maintain a small platform-detection table in `reference/bundle/` and emit the right command. For now, generic instruction (see Understand-Anything's README).

---

## 3. Multi-language template installation

**Status:** wiki-init flattens `reference/bundle/templates/<lang>/*.md` into the target's `templates/*.md` (without the language subdirectory). Only one language per wiki.

**Open:** should the researcher be able to install both `en` and `ko` templates side-by-side (with name suffix or subdirectory)?

**Proposed:** no. One language per wiki keeps `wiki-log`'s template lookup unambiguous. If the researcher wants to switch, they re-init in a different repo or manually copy templates from `skills/wiki-init/reference/bundle/templates/<other-lang>/`.

---

## 4. `ARCHITECTURE.md` in target repo

**Status:** NOT created by default. The target repo relies on the upstream design document in the skill-set repo.

**Open:** should there be an `--include-architecture` flag for researchers who want a self-contained workspace?

**Proposed:** yes for v1.x. Defer until a researcher actually asks. The default of "rely on upstream" keeps target repos lean and avoids design-doc drift between many wiki workspaces.

---

## 5. Idempotency edge case — what if the researcher renamed `CLAUDE.md`?

**Status:** wiki-init checks for `CLAUDE.md` by exact name. If the researcher renamed it (e.g., to `CONSTITUTION.md`), wiki-init would think the workspace is uninitialized and try to write a new `CLAUDE.md`.

**Open:** should wiki-init also detect by content (e.g., look for the constitution heading)?

**Proposed:** no. CLAUDE.md is the canonical filename in the entire skill set; renaming it breaks every other skill anyway. Detect by name only; document the convention.
