You are about to help with **ResearchWiki**. Use this briefing as the load-bearing context for everything you do in or around this codebase. If a later instruction conflicts with what is written here, ask before deviating.

## 1. What ResearchWiki is

A skill set for LLM coding agents that maintains a **living research journal** alongside a researcher's code. The journal is a markdown wiki that links across the researcher's three accumulating bodies of knowledge — code, papers, and experiments — and updates as the research evolves.

ResearchWiki is a vertical application of the [Codified Context](https://arxiv.org/abs/2602.20478) framework to research workflows. Citations to Codified Context and to [Understand-Anything](https://github.com/Lum1104/Understand-Anything) are mandatory in any derivative work.

## 2. The problem it solves

Researchers accumulate knowledge in three forms at once:

- Code they write
- Papers they read
- Experiments they run

The *connections* between these — which code implements which paper's idea, which experiment tested which hypothesis, which design decision resolved which contradiction — are where the real understanding lives. They are also where existing tools fail: Zotero handles papers, MLflow handles experiments, Understand-Anything handles code, and none of them handle the crossings. The connections live in the researcher's head, get forgotten, and have to be rediscovered.

ResearchWiki is the glue.

## 3. Architecture: three layers

| Layer | Path | Role | Cadence | Owner |
|---|---|---|---|---|
| Wiki Layer | `wiki/` | **Interpretation.** Researcher's journal: concept pages, paper summaries, experiment writeups, design decisions, daily log. Markdown with frontmatter. | Continuous, daily | Human-authored, LLM-assisted |
| Index Layer | `index/` | **Fact.** Lightweight snapshot of the codebase: file tree, function/class signatures, recent diffs, active experiments. Plus immutable per-run snapshots and audit reports. | Daily, fast | Regenerated by tool |
| Deep Analysis Layer | `deep/` | **Fact, expensive.** Rich code knowledge graph from Understand-Anything. | Weekly or at milestones | Regenerated by tool |

A fourth path, `raw/`, holds source materials (papers, experiment logs) and is human-managed.

The **fact / interpretation separation** between Wiki and (Index + Deep) is principle #1. It is not negotiable. Never merge them. Never silently move content from one to the other.

## 4. The eight skills (five MVP + two retrieval + one remediation)

**MVP — without these, the workspace cannot complete its life-cycle.**

| Skill | Cadence | What it does | Writes to |
|---|---|---|---|
| `wiki-init` | Once per project | Bootstraps the workspace: directories, `CLAUDE.md`, `research-wiki.config.yaml`, default templates, seed wiki pages. Idempotent. | Target repo root + scaffolding |
| `wiki-log` | Multiple times per day | The researcher's primary skill. Loads a template (experiment / paper_reading / design_decision / free_form), fills it conversationally, auto-links against existing content, **suggests new concept stubs for unmatched noun phrases**, appends to `wiki/log.md`, updates `wiki/index.md`. | New page in `wiki/`, frontmatter back-refs on linked pages, optional new concept stubs |
| `wiki-sync` | Daily ("morning ritual") | Regenerates the Index Layer. Detects stale `refs.code` in wiki pages and flags them via frontmatter and `wiki/questions.md`. Also produces `index/reverse_refs.json` (code path → referencing wiki pages). | `index/snapshots/sync_*.md`, `index/signatures.json`, `index/reverse_refs.json`, frontmatter flags |
| `wiki-deepscan` | Weekly / milestone | Runs Understand-Anything. Refreshes the deep knowledge graph. Optionally seeds stub pages for newly significant components. | `deep/knowledge-graph.json`, `deep/last-scan.yaml`, stub `wiki/` pages (create-only) |
| `wiki-lint` | On-demand audit | Eight mechanical checks: frontmatter schema, broken links, file-existence of `refs.code`, speculation density (>30% by default), persistent stale flags (>7 days), confidence-conflict contradictions, orphan pages. **Reports findings; never fixes them. Never modifies a wiki page (body or frontmatter).** | `index/audits/lint_*.md`, append-only on `wiki/questions.md` and `wiki/discrepancies.md` |

**Retrieval extensions — read-only, depend on MVP artifacts, materially improve daily ergonomics.**

| Skill | Cadence | What it does | Writes to |
|---|---|---|---|
| `wiki-query` | On-demand | Natural-language search over wiki contents. Returns ranked page paths + extractive snippets to stdout, with `⚠ stale: …` badge prefix on pages with unresolved `stale: true` flags. MVP backend is BM25-style lexical (whitespace + identifier tokenizer, ko/en mix); embedding-based semantic search is reserved for v2. **Never composes an answer or summarizes — pointers + verbatim spans only.** | nothing (stdout) |
| `wiki-recall` | Periodic / pre-milestone | Surfaces stale-but-relevant pages by intersecting recent `wiki/log.md` activity refs with old-page frontmatter refs. Weighted overlap score; frontmatter-only (does not parse bodies — that is `wiki-query`'s job). **Pointers + explicit overlap evidence; never summarizes.** | nothing (stdout) |

**Remediation — the *one* skill that legitimately edits wiki page bodies, under strict per-occurrence researcher approval.**

| Skill | Cadence | What it does | Writes to |
|---|---|---|---|
| `wiki-fix-stale` | On-demand maintenance, typically prompted by `wiki-sync`'s nag | Walks the researcher through unresolved stale refs (frontmatter + optional body link rot mentions). Per occurrence, presents four mechanical options (replace symbol, wrap-deprecated, delete sentence, skip). Applies only the chosen literal transformation; never composes new prose. After all refs in a page are addressed (no skips), auto-clears the `stale: true` frontmatter flag. **Closes the body / frontmatter decoupling gap — the design's most damaging operational hazard — with strict P3 compliance (researcher-initiated + per-occurrence approval).** | wiki page bodies (per-edit approved), wiki/ frontmatter, wiki/log.md |

## 5. The eight principles (read these before doing anything)

Defined in `ARCHITECTURE.md §1.4`, restated as constitution in `CLAUDE.md`. The four most load-bearing:

- **P1 — Fact and interpretation are separate.** `wiki/` is interpretation. `index/` and `deep/` are facts. Never merge them.
- **P3 — Propose, do not mutate interpretation.** You may freely regenerate `index/` and `deep/`. You may **not** silently rewrite anything in `wiki/`. When you detect that a wiki page is stale or contradictory, update *only the frontmatter* (and only if it concerns fact-layer references), and append a note to `wiki/questions.md`. The researcher decides whether and how to update the body.
- **P7 — Every claim has provenance.** Frontmatter: `authored_by ∈ {human, llm, hybrid}`, `confidence ∈ {verified, inferred, dynamic}`. Inline content: quote sources, or tag `[speculation]` if ungrounded.
- **P8 — Analysis yes, speculation no.** *(the hardest one)* You may summarize what sources contain, describe observable relationships, juxtapose for contrast. You may **not** guess intent the source does not state, extend implications a paper does not make, propose causal explanations the logs do not support, or critique design unprompted. When in doubt: prefer a question, prefer the `[speculation]` tag, or omit. **Violating P8 is the most damaging thing you can do** — it slow-poisons the knowledge base with plausible-but-wrong claims that may not get caught for weeks.

The remaining four (P2 low daily friction, P4 configuration over convention, P5 markdown-first, P6 Korean default) shape the operational ergonomics; consult `ARCHITECTURE.md §1.4` when they matter.

## 8. How to engage with this codebase

1. **Read `CLAUDE.md` fully first.** It is the constitution. Skipping it is a protocol violation. (See `CLAUDE.md §8` for the full session-start checklist.)
2. **Glance at `ARCHITECTURE.md`'s table of contents** if your task touches design or open questions.
3. **Use existing skills, don't freelance.** If a task fits `wiki-log`, don't hand-author a wiki entry. If it fits `wiki-sync`, don't ad-hoc grep the codebase. The skills exist precisely to keep behavior consistent across sessions.
4. **Default to less prose, more questions.** When uncertain, append to `wiki/questions.md` rather than write speculation into a page body. The researcher reads questions; they cannot easily catch a confident wrong sentence buried in prose.
5. **Tag every claim's provenance.** Frontmatter for metadata; inline `[speculation]` or `[unverified]` for ungrounded statements.
6. **Never edit a wiki page body without explicit researcher approval.** The frontmatter `refs:` block is the only zone open for LLM-initiated edits, and only for fact-layer reference updates (per `CLAUDE.md §3` directory contract).
7. **If you find yourself wanting to write more than the source supports, stop.** Prefer an honest gap to a confident-but-wrong claim. P8 is the principle that protects the long-term value of the knowledge base.
