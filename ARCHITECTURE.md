# ResearchWiki — Architecture

> **Status:** Draft v0.2
> **Purpose:** Design reference for the ResearchWiki skill set — a vertical application of the [Codified Context](https://arxiv.org/abs/2602.20478) framework to the research domain.
> **Audience:** Contributors, future maintainers, and LLM agents operating on this repository.

---

## Part 1 — Foundation

### 1.1 What is ResearchWiki

ResearchWiki is a set of LLM-agent skills that help a researcher maintain a **living research journal** during active research — especially when that research involves both writing new code and synthesizing ideas from papers and experiments.

It is not a note-taking app, not a code analysis tool, and not a RAG system. It is a **skill set for an LLM coding agent** (primarily Claude Code) that lets the agent *maintain a structured knowledge base on behalf of the researcher*, while the researcher focuses on curation, questioning, and actual research work.

The output of using ResearchWiki is a directory of interlinked markdown files plus a lightweight code index, which together function as the researcher's externalized memory. The knowledge base is meant to be read in Obsidian (or any markdown tool) and committed to git like any other research artifact.

### 1.2 Where it fits — relation to existing work

ResearchWiki sits at the intersection of three recent ideas.

**Codified Context (arXiv:2602.20478)** — the foundational framework. Proposes that unstructured developer knowledge should be preprocessed into structured context for AI coding agents, organized in a three-tier architecture: Hot Memory (CLAUDE.md as constitution), specialist agents (Tier 2), Cold Memory (knowledge base). ResearchWiki is a **vertical application** of this framework to the research domain.

**LLM Wiki pattern (conceptual)** — the idea that an LLM should incrementally build and maintain a persistent, interlinked wiki as new sources arrive, rather than re-deriving everything at query time. ResearchWiki adopts this pattern for the **interpretation layer** of the knowledge base.

**Understand-Anything (github.com/Lum1104/Understand-Anything)** — a tool that analyzes a codebase with a multi-agent pipeline and produces an interactive knowledge graph. ResearchWiki uses it (optionally) as the **deep analysis layer** for codebases, but does not depend on it for daily operation.

ResearchWiki's contribution is the **integration architecture**: how to make these three layers — handwritten constitution, incremental wiki, automated code graph — cooperate without drifting apart, in the specific context of a researcher whose code changes daily.

### 1.3 Target user and non-users

**Target user:** A researcher actively coding while also reading papers and running experiments — the "mixed-mode" researcher.

- Writes new code daily (not just reading an existing codebase)
- Reads papers at an irregular but regular cadence
- Runs experiments whose results need to be recorded in connection with code versions
- Values a structured knowledge base but has not been able to maintain one by hand
- Uses an LLM coding agent as part of daily work
- Works with markdown and git already

**Explicit non-users (initial version):**

- Pure literature researchers with no codebase → overkill; plain LLM Wiki is enough
- Teams onboarding onto an existing stable codebase → Understand-Anything alone is enough
- Researchers who do not use LLM agents → the maintenance burden returns
- Users who want a polished UI → this is a markdown + CLI tool, not an app

### 1.4 Core design principles

These principles override any specific design choice later in the document. When in conflict, the principle wins.

**P1 — Separation of fact and interpretation.**
The knowledge base keeps two kinds of content distinct: *facts* (what functions exist, what calls what, what file a module lives in) and *interpretations* (why this architecture was chosen, what this experiment means, what this paper's claim implies). Facts are generated by tools and can be overwritten safely. Interpretations are authored with human input and must not be overwritten by automation.

**P2 — Low daily friction, high weekly depth.**
Daily-use commands are few and easy to remember. Deeper, more expensive operations (full codebase scans, lint passes) live in separate, less-frequent commands. The daily loop is sustainable without conscious effort.

**P3 — Propose, do not mutate interpretation.**
Skills may freely create, modify, and delete *fact-layer* artifacts. Skills must never silently rewrite the researcher's *interpretation-layer* writing. When skills detect staleness or contradiction in interpretation, they propose a change and log the proposal, but leave the final edit to the researcher.

**P4 — Configuration over convention, for anyone but you.**
Every directory name, template field, and schema choice is configurable via a single config file. Sensible defaults exist, but contributors can bend the tool to their research style without editing skill code.

**P5 — Claude Code first, other platforms best-effort.**
The skill set is authored for Claude Code's SKILL.md format and its native conventions. Skill bodies prefer portable patterns where it costs little, but we do not sacrifice Claude Code quality to support other platforms. Compatibility with Codex / Cursor / OpenCode is a welcome side effect, not a design constraint.

**P6 — File-based, git-native storage.**
No database. Everything is markdown, YAML, or JSON. The knowledge base is a git repository.

**P7 — Explicit uncertainty.**
Every inferred link or generated claim is tagged with its provenance and confidence level (e.g., `verified`, `inferred`, `dynamic`, `user-authored`). A reader of the wiki can always tell where a statement came from.

**P8 — Analysis yes, speculation no.** *(the "no-guessing" principle)*
Skills produce analysis based on what is *present* in the sources. Skills do not produce speculation about what is absent, implied, or intended but unstated.

*Permitted:*
- Summarization of content explicitly present in code, papers, or logs
- Description of observable relationships between artifacts (function A calls function B; paper X is cited by paper Y)
- Juxtaposition that surfaces similarities, differences, or contradictions between sources
- Judgments that can be stated as `X is Y` rather than `X seems like Y`

*Forbidden:*
- Inferring developer intent that the code does not state ("the author probably meant...")
- Extending implications a paper does not explicitly make ("the authors seem to imply...")
- Asserting causes for experiment outcomes that the logs do not establish ("this failure was likely caused by...")
- LLM-authored design critique that the researcher has not requested or seeded

*Gray-zone handling:*
When the distinction is unclear, the skill must (a) mark the statement with a `[speculation]` or `[unverified]` tag, and (b) prefer a question form over a claim ("intent here is not clear from the code — confirm with author").

This principle is the single strongest defense against slow poisoning of the knowledge base by plausible-but-wrong LLM output.

### 1.5 Language policy

- **Default documentation language:** Korean
- **Session-following rule:** An LLM session may produce output in whatever language dominates the current session. A session in English produces English entries; a session in Korean produces Korean entries.
- **Mixed-language wiki:** Wiki pages may exist in either language. Cross-links work across languages. No automatic translation.
- **Privacy assumption:** All research notes are assumed to be *public-safe content*. Skills need not implement redaction or privacy filters. If a researcher has private content, they keep it outside the wiki.

## Part 2 — Architecture

### 2.1 The three layers

ResearchWiki's Cold Memory is organized into three layers, each with different update cadence, different owners, and different trust levels.

```
+----------------------------------+
|  Wiki Layer (Interpretation)     |  ← human + LLM, daily, low automation
|  wiki/                           |
+-----------------↕----------------+
|  Index Layer (Light Facts)       |  ← scanner + LLM summary, daily
|  index/                          |
+-----------------↕----------------+
|  Deep Analysis Layer (Heavy)     |  ← Understand-Anything, weekly+
|  deep/                           |
+----------------------------------+
            ↓
      Raw sources
      (code, papers, experiment logs)
```

**Wiki Layer.** The researcher's interpreted understanding. Entries for concepts, design decisions, paper summaries, experiment write-ups. LLM assists with authoring but cannot overwrite without permission (P3). This is where meaning lives.

**Index Layer.** A lightweight, regenerable record of "what currently exists in the repository." File tree, function and class signatures, recent diffs, active experiment list. Cheap to regenerate (seconds). Each `wiki-sync` run writes a new **immutable timestamped snapshot** under `index/snapshots/sync_YYYYMMDD_HHMM.md` (chronological history, never overwritten) and overwrites `index/signatures.json` (machine-readable, consumed by `wiki-log` auto-link). This is the fact surface the Wiki Layer links *down to*, and it doubles as the "progress report" baseline — every snapshot is a self-contained markdown that can be read on its own.

**Deep Analysis Layer.** Understand-Anything output — full knowledge graph, layer classification, guided tours. Expensive to regenerate (minutes). Overwritten each `wiki-deepscan` run. Provides the richer fact surface for cross-cutting questions.

The **Wiki Layer contains interpretation**; the **Index and Deep layers contain fact**. P1 is structurally enforced by keeping these in separate directories with separate update rules.

### 2.2 Directory structure

```
<project-root>/
├── CLAUDE.md                       # Hot Memory — constitution for the skill set
├── research-wiki.config.yaml       # User-facing configuration
├── wiki/                           # Wiki Layer (interpretation)
│   ├── index.md                    # Content catalog (LLM-maintained)
│   ├── log.md                      # Chronological activity log
│   ├── concepts/                   # Concept pages
│   ├── papers/                     # Paper summaries
│   ├── experiments/                # Experiment write-ups
│   ├── decisions/                  # Design decisions
│   ├── questions.md                # Open questions / things to verify
│   └── discrepancies.md            # Contradictions between sources
├── index/                          # Index Layer (light facts)
│   ├── snapshots/                  # Immutable timestamped snapshots, one per wiki-sync run
│   │   └── sync_YYYYMMDD_HHMM.md   #   Self-contained markdown (state + diff since previous)
│   └── signatures.json             # Function/class signatures (machine-readable, overwritten)
├── deep/                           # Deep Analysis Layer (optional)
│   ├── knowledge-graph.json        # Understand-Anything output
│   └── last-scan.yaml              # Timestamp + scope of last deepscan
└── raw/                            # Raw sources (optional; may live elsewhere)
    ├── papers/                     # Raw PDFs, clipped articles
    └── experiments/                # Raw experiment logs
```

All paths above are *defaults* overridable in `research-wiki.config.yaml` (P4).

### 2.3 Link schema and frontmatter

Every wiki page carries YAML frontmatter with standardized fields. The schema is versioned so skills can evolve it.

```yaml
---
schema_version: 1
type: concept | paper | experiment | decision | other
created: 2026-04-23
updated: 2026-04-23
tags: [optional, free-form]

# Cross-references (the backbone of the wiki)
refs:
  code:
    - path: src/auth/login.ts
      symbol: loginUser
      confidence: verified   # verified | inferred | dynamic
  papers:
    - id: original-2024
  concepts:
    - transformer-attention
  experiments:
    - exp-2026-04-22-lr-sweep

# Provenance (P7)
authored_by: human | llm | hybrid
source_sessions: [2026-04-23-session-1]
---
```

**Confidence levels:**
- `verified` — directly observable in the Index or Deep layer (e.g., function `loginUser` exists at `src/auth/login.ts` per Index Layer scan)
- `inferred` — stated in a comment, docstring, or document; not structurally verifiable
- `dynamic` — runtime-only relationship (plugin, callback, reflection) — unverifiable without execution

**Authorship tags:**
- `human` — researcher wrote this page
- `llm` — skill generated this page (subject to stricter display rules; see P3, P8)
- `hybrid` — both contributed; most common

### 2.4 The config file

A single `research-wiki.config.yaml` governs every user-tunable behavior. Full specification lives in `docs/CONFIG.md` (TBD). Minimum fields:

```yaml
schema_version: 1
language:
  default: ko            # fallback when session language is ambiguous
  follow_session: true   # use session-dominant language

paths:
  wiki: wiki/
  index: index/
  deep: deep/
  raw: raw/

log_templates:
  experiment: default
  paper_reading: default
  design_decision: default
  free_form: default
  # Each may point to a custom template under templates/

deepscan:
  tool: understand-anything    # or: none
  cadence_hint: weekly         # informational; tool does not auto-run

lint:
  strict_mode: false           # if true, refuse to write pages with [speculation] tags
```

### 2.5 Provenance and the no-guessing principle

The combination of §1.4 P7 (explicit uncertainty) and P8 (no speculation) is the trust foundation of the wiki. Operationally:

- Every sentence in an LLM-generated page that states a causal claim, an intent claim, or an implication must be either (a) traceable to a source in `refs:`, or (b) tagged `[speculation]` inline.
- Pages where more than 30% of claims are `[speculation]`-tagged are flagged by `wiki-lint` as low-trust and recommended for researcher review.
- A `strict_mode` config option (see §2.4) rejects writes of `[speculation]`-heavy content entirely.

### 2.6 How a code change propagates

To make P1 and P3 concrete, here is the flow when the researcher changes code:

1. Researcher modifies `src/trainer.py`, commits.
2. Next time the researcher runs `wiki-sync`:
   - Index Layer re-scans. `index/signatures.json` is overwritten with the latest symbols.
   - A new immutable snapshot `index/snapshots/sync_YYYYMMDD_HHMM.md` is written. It contains the current state (file tree, modules, active experiments) **and** a "Changes since previous snapshot" section describing what symbols were added/removed/changed since the last snapshot.
3. `wiki-sync` also runs a **stale-link scan** over the Wiki Layer:
   - For each wiki page with `refs.code` entries, check if the referenced symbols still exist.
   - If not, mark the reference `stale: true` in the frontmatter (fact-layer change, allowed).
   - Append a note to `wiki/questions.md`: "Page X references removed symbol Y — update needed."
   - **Do not silently rewrite the wiki page itself** (P3).
4. Researcher decides what to do — manually update the page, or ask the agent to update it explicitly.
5. On the next `wiki-deepscan` (weekly or on-demand), the Deep Analysis Layer refreshes; Wiki Layer gets richer code links for new material.

This design guarantees that interpretation never gets silently corrupted by code churn.

## Part 3 — Skill Set

### 3.1 Skill set — five MVP + two retrieval + one remediation

The five skills below form the MVP — without all of them, ResearchWiki cannot complete the full life-cycle (init → log → sync → deepscan → lint).

| Skill | Frequency | Tier | Writes to |
|---|---|---|---|
| `wiki-init` | Once per project | Init | everything (seed) |
| `wiki-log` | Multiple times per day | Daily | wiki/, log.md |
| `wiki-sync` | ~Once per day | Daily | index/, frontmatter of wiki/ |
| `wiki-deepscan` | Weekly+ or on milestone | Weekly | deep/, wiki/ stubs |
| `wiki-lint` | On-demand | Audit | questions.md, discrepancies.md, reports |

Two additional **retrieval extensions** were added after the MVP closed. They are read-only and depend on artifacts the MVP skills produce; they do not write to the wiki and are not strictly required for the life-cycle, but materially improve daily ergonomics for a long-running wiki.

| Skill | Frequency | Tier | Writes to |
|---|---|---|---|
| `wiki-query` | On-demand | Read-only | nothing (stdout) |
| `wiki-recall` | Periodic | Read-only | nothing (stdout) |

One **remediation** skill closes the design's most damaging operational hazard — the body / frontmatter decoupling, where `wiki-sync` flags stale refs in frontmatter but the body still claims things about removed symbols. It is the *only* skill that legitimately edits wiki page bodies after creation, and it does so under the strict P3 condition CLAUDE.md §3 permits ("only with explicit researcher approval or researcher-initiated request"): researcher-initiated invocation plus per-occurrence approval of every body edit.

| Skill | Frequency | Tier | Writes to |
|---|---|---|---|
| `wiki-fix-stale` | On-demand maintenance | Remediation | wiki/ bodies (per-edit approved), frontmatter, log.md |

Each skill has its own full specification in `skills/<skill-name>/SPEC.md`.

### 3.2 Skill boundaries

The skills are designed so that each one has a single clear responsibility and does not duplicate the work of another:

- `wiki-init` is the **only** skill that creates the initial directory layout and config. Other skills assume these exist.
- `wiki-log` is the **only** skill that writes new content to the Wiki Layer based on researcher input. Other skills may update metadata but not author new interpretation.
- `wiki-sync` is the **only** skill that regenerates the Index Layer (including the reverse-ref index `index/reverse_refs.json`).
- `wiki-deepscan` is the **only** skill that invokes the external Understand-Anything tool.
- `wiki-lint` is the **only** skill that produces audit reports; it does not itself fix what it finds.
- `wiki-query` is the **only** skill that performs natural-language retrieval over wiki page contents. It is read-only and never writes a file.
- `wiki-recall` is the **only** skill that surfaces stale-but-relevant pages by intersecting recent log activity with old-page frontmatter refs. It does not parse page bodies (that is `wiki-query`'s job).
- `wiki-fix-stale` is the **only** skill that edits wiki page bodies after creation. The legitimacy rests on three protections: (1) researcher-initiated invocation, (2) per-occurrence approval of every individual edit, (3) only four pre-defined mechanical transformations (replace symbol, wrap-deprecated, delete sentence, skip) — no LLM-authored prose. If any of the three protections is weakened, the skill ceases to be P3-compliant.

### 3.3 Daily / weekly / milestone workflow

```
First day:         wiki-init        (seed the project once)
Each morning:      wiki-sync        (refresh index, catch stale links)
Throughout day:    wiki-log         (record experiments, readings, decisions)
Weekend / milestone:
                   wiki-deepscan    (refresh Understand-Anything view)
                   wiki-lint        (audit; triage findings)
```

### 3.4 Template system

`wiki-log` is driven by templates defined in `research-wiki.config.yaml` and stored under `templates/`. The MVP ships with four default templates: `experiment`, `paper_reading`, `design_decision`, `free_form`. Each template defines (a) required frontmatter fields, (b) a set of prompts for the researcher, and (c) auto-link rules (which related items to scan and link).

Full template specification lives in `docs/TEMPLATES.md`.

### 3.5 Implementation strategy — Class A (pure code) vs Class B (LLM hybrid)

The eight skills split cleanly along *what kind of computation each requires at runtime*:

**Class A — pure code (7 skills).** `wiki-init`, `wiki-sync`, `wiki-deepscan`, `wiki-lint`, `wiki-query`, `wiki-recall`, `wiki-fix-stale`. Implemented as standard Python under `src/researchwiki/`. Each exposes a `run_<skill>` library function and a CLI entry point. Reasoning is mechanical: AST scanning, BM25 ranking, frontmatter manipulation, lexical pattern matching, scripted prompts. The corresponding `SKILL.md` is a thin wrapper that tells an LLM "to do X, invoke `wiki-<skill>` via Bash" — the LLM is a router, not a reasoner. P3/P8 are enforced by code (the skills literally cannot violate them).

**Class B — LLM hybrid (1 skill).** `wiki-log`. The mechanical operations (template parsing, frontmatter assembly, code-symbol lookup, atomic writes, bidirectional back-refs, concept stub creation) live in `src/researchwiki/log.py` behind a 5-subcommand CLI (`inspect`, `lookup-symbols`, `find-pages`, `find-amend-target`, `run`). The conversational layer that drives the entry's creation — paraphrasing italic template guides into natural questions, detecting P8 hedge/causal/intent markers and routing them through the three-route flow, extracting identifier candidates and noun phrases from researcher prose, choosing the summary line, handling refusal cases — is the LLM's responsibility, guided by a rich `reference/` library (`p8-detection.md`, `conversational-style.md`, `auto-link-extraction.md`, `refusal-patterns.md`, `templates-deep-dive.md`). The CLI is the LLM's *toolkit*, not its script.

**Why the split is one-skill-deep.** wiki-log is the only skill that, by its nature, requires runtime *interpretation of a researcher's free-form prose*. The other seven operate on structured artifacts (code symbols, frontmatter, log entries, slug matches) where mechanical pattern-matching is sufficient and LLM involvement would only add nondeterminism. Pushing wiki-log into Class A would have required either (a) making the conversation a multi-prompt scripted form (ergonomics regression — the design driver for wiki-log is *fluid conversation*) or (b) fixing a single LLM-implemented heuristic into code (P8 detection on Korean prose does not collapse into a regex). The hybrid keeps each layer doing what it is best at.

**Testing implication.** Class A skills are testable end-to-end with pytest — every assertion can pin a deterministic outcome to a specific input. Class B's mechanical layer is similarly testable; integration tests (`tests/integration/test_scenario_wiki_log_full.py`) hand-author the payload an LLM would have produced and verify the CLI flow. The *LLM-side reasoning* (P8 detection accuracy, conversational fluidity, extraction precision) is not exercised by automated tests; doing so would require a separate test infrastructure (real Claude API calls with cost / nondeterminism management, HTTP-cassette replay, or scripted LLM mocks) that is out of scope for v0.1. Manual smoke runs are the current discipline.

### 3.6 Extension points

Future skills that fit this architecture without violating P1~P8:

- `wiki-export` — compile a slice of the wiki into a paper draft, slides, or report
- `wiki-triangulate` — surface patterns that involve simultaneously code + paper + experiment (the triple-link signature of mixed-mode research)

These are explicitly out of scope for MVP. (`wiki-query` was previously listed here; it is now implemented and documented in §3.1.)

### [2026-04-25] Two retrieval extensions + wiki-sync reverse index + wiki-log concept stubs

This batch was added the same day after the design-overview pass, in response to the question "is the MVP enough to be called a knowledge graph + accumulating analysis system?" The honest answer was *structurally yes, but the system has no query layer and no surfacing layer*. These extensions close those gaps without violating P3 / P8.

- **`wiki-sync` now produces `index/reverse_refs.json`.** Pure inversion of declared `refs.code` frontmatter — no body parsing, no inferred bindings. Rationale: code-centric researchers need "who references this path before I refactor?" and the data already exists; building it during the same wiki-frontmatter scan that drives stale-link detection is essentially free. The skill boundary is preserved: wiki-sync still does not parse bodies for code references (that gap is documented in `wiki-sync/SPEC.md` open questions).

- **New `wiki-query` skill — natural-language retrieval over the wiki corpus.** Read-only, BM25-style lexical scoring over a whitespace + identifier tokenizer (handles `snake_case`, `camelCase`, `kebab-case`, ko/en mix). Returns ranked page paths + extractive snippets to stdout. Embedding-based semantic search is reserved as a v2 extension behind `query.backend: hybrid` config. Rationale: grep is too narrow (requires exact wording recall), `wiki/index.md` is too coarse (flat catalog); the missing middle is keyword-graded retrieval. P8 alignment: the skill returns *pointers + extracted spans*, never composes an answer or summarizes — every snippet is verbatim.

- **New `wiki-recall` skill — surfacing stale-but-relevant pages.** Read-only, frontmatter-only (does not parse bodies — that is `wiki-query`'s job). Computes weighted overlap between (i) `refs:` declared in `wiki/log.md` entries within the lookback window and (ii) `refs:` declared in pages older than the staleness threshold. Default weights: `refs.code = 2.0`, `refs.concepts = 1.5`, `refs.papers = 1.0`, `refs.experiments = 1.0`. Excludes wiki-deepscan stubs (`authored_by: llm` + empty body) by default. Rationale: long-running wikis accumulate forgotten material that becomes relevant again when work circles back to similar refs; surfacing it mechanically (no summary, no recommendation) is the maximally cautious form of "memory refresh".

- **`wiki-log` now suggests concept stubs for unmatched concept-sounding noun phrases.** Extension to step 3 of the auto-link rules: phrases that fail the existing exact-slug-match are routed to a separate "create new concept stub?" batch the researcher can approve, reject, or edit per phrase. Approved stubs are created with frontmatter only and a single italicized "Add interpretation here" line — wiki-log never writes prose about what the concept "is for". Stubs carry `seeded_by: wiki-log` and `seed_context:` so downstream skills can recognize them. Gated by `log.suggest_concept_stubs:` config (default `true`). Rationale: dropped concepts compound — an entry mentioning "rotary embedding" without an existing page silently loses the cross-link forever; this extension catches the loss the moment it would happen.

- **Skill set count revised: five MVP + two retrieval extensions.** `ARCHITECTURE.md §3.1` updated; `README.md` skill table extended; `prompts/project-overview.md` table extended. **Note:** `CLAUDE.md §4` still lists five skills — updating the constitution is a deliberate governance act and was deferred to explicit researcher decision. Two unresolved follow-ups: (a) propagate the count to `CLAUDE.md` and `wiki-init/reference/CLAUDE.md`; (b) refine `wiki-lint` Check #8 (orphan) to honor `seeded_by:` markers with a grace period — a stub is *expected* to be orphan at creation time, so flagging it on the next audit is noise.

### [2026-04-25] Body / frontmatter decoupling closure: wiki-fix-stale + wiki-sync body link rot + wiki-query stale badge

This batch was added the same day after a critical evaluation of the demo artifacts. Five flaws were identified in the accumulated knowledge base; the most damaging — *the body still says false things while the frontmatter quietly admits the truth via `stale: true`* — was selected as the priority closure target. The fix is a coordinated three-way change rather than a single skill, because the gap exists at the seam between three responsibilities: detection (sync), surfacing (query), and resolution (a new skill).

- **New `wiki-fix-stale` skill — researcher-initiated body remediation.** Walks the researcher through unresolved stale refs (frontmatter + optional body link rot mentions) one occurrence at a time. For each occurrence presents four mechanical options: replace symbol with researcher-supplied name, wrap with `[deprecated YYYY-MM-DD]` tag, delete the affected sentence, or skip. Applies only the chosen literal transformation; never composes new prose. After all refs in a page are addressed (no skips), auto-clears `stale: true` from frontmatter. Rationale: P3 ("propose, do not mutate interpretation") forbids *silent* body edits but explicitly permits *researcher-initiated and per-occurrence-approved* edits (CLAUDE.md §3 directory contract). wiki-fix-stale is the precise scenario the carve-out was reserved for. It is the *one* skill in the set that legitimately edits wiki page bodies.

- **`wiki-sync` extension — opt-in body link rot scan.** When `--scan-body` is set (or `config.sync.body_link_rot.enabled: true`), wiki-sync scans body prose for identifier-shape tokens whose names no longer appear in `index/signatures.json`, and records misses as `body_stale_mentions:` in frontmatter. Heuristic — false positives possible (English nouns matching PascalCase, comments containing module references) — so each entry implicitly carries `[unverified]` semantic that downstream consumers (notably `wiki-fix-stale`) must surface to the researcher. Rationale: frontmatter `stale: true` only catches refs the wiki *declared* in `refs.code`; many wiki pages mention code symbols in prose without listing them in frontmatter. The body link rot scan catches that gap. Opt-in because it's the slowest part of a sync run and the false-positive surface deserves an explicit researcher decision before being injected into frontmatter.

- **`wiki-sync` extension — end-of-run nag.** After the run summary, if pages have unresolved `stale: true` flags older than `config.sync.nag_after_days` (default 7), append a one-line reminder pointing at `wiki-fix-stale`. Suppressible via `--no-nag`. Rationale: the body / frontmatter decoupling is corrosive precisely because it is invisible — without an active reminder, accumulated stale flags become wiki noise the researcher tunes out. The nag is *surfacing*, not enforcement: wiki-sync does not invoke wiki-fix-stale, does not retry, does not block.

- **`wiki-query` extension — stale badge on results.** Result pages with unresolved `stale: true` flags are prefixed with `⚠ stale: N refs unaddressed for Md`. Stale pages are *not filtered out* — the researcher may still need to read them — but the badge surfaces the body / frontmatter decoupling risk at the moment the researcher is about to consume the page's content. Configurable via `query.stale_warnings:` (default `true`); `--no-stale-warnings` overrides. Rationale: P7 mandates explicit uncertainty; surfacing it where the researcher actually consumes search results is the highest-leverage intervention point.

- **Skill set count revised again: five MVP + two retrieval + one remediation = eight.** `ARCHITECTURE.md §3.1` updated; `README.md` skill table extended; `prompts/project-overview.md` table extended; `wiki-init/SKILL.md` description extended. Outstanding follow-ups carried over from prior batch: (a) propagate the count to `CLAUDE.md §4` and `wiki-init/reference/CLAUDE.md` (still deferred — constitution change requires explicit researcher decision); (b) `wiki-lint` Check #8 orphan refinement to honor `seeded_by:` markers (still pending).

- **Closure check.** With `wiki-fix-stale` available and `wiki-sync` / `wiki-query` surfacing the gap aggressively, the body / frontmatter decoupling becomes a *recoverable* state rather than a *silent corrosion*. The wiki can still develop body-lies between fixes — the researcher must still invoke wiki-fix-stale to actually fix them — but the surface is now visible (badge + nag) and the remediation path is explicit (skill exists). Of the five flaws identified in the demo evaluation, this batch closes the one rated "most dangerous" plus reduces the slow-poisoning surface area materially.

### [2026-04-26] Reference/ pattern refactor — overengineering reduction

This pass was triggered by a critical question — *had the design become overengineered?* — and an honest yes-in-specific-places diagnosis. The most concrete cost was **runtime context budget**: the previous pattern stuffed worked examples, full failure-mode catalogs, open-question lists, and verbose P8-enforcement subsections directly into each `SKILL.md`, growing them to 232–640 lines per skill. Every invocation reloaded all of that.

The refactor introduced a **per-skill `reference/` subdirectory** that holds supplementary material the LLM does not need at runtime, splitting each skill into:

```
skills/<name>/
├── SKILL.md       ← LLM runtime prompt (~100–180 lines)
├── SPEC.md        ← human design document (~50–80 lines)
└── reference/
    ├── examples.md
    ├── failure-modes.md
    ├── open-questions.md
    └── consumed-config.md
```

- **Aggregate effect.** Total `SKILL.md` payload across all 8 skills: **3015 → 994 lines (−67%)**. Each skill individually saw 50–77% reduction. Information loss: 0 (everything moved to `reference/`, nothing deleted). Total project size grew (~7200 lines) because the move expanded some sections and added structure (`consumed-config.md` did not exist before), but the *runtime-loaded* portion shrank ~3×.

- **`wiki-init/reference/bundle/` carve-out.** `wiki-init` is the only skill that ships *runtime assets* (the constitution, default config, default templates) that get copied to the target repo at init time. Those moved from `reference/` (where they used to live alongside the docs files we were now adding) into `reference/bundle/` so the bundle assets and the documentation supplements coexist with clean visual separation. All `wiki-init/SKILL.md` references to `reference/CLAUDE.md`, `reference/research-wiki.config.yaml`, and `reference/templates/<lang>/` were updated to the new `reference/bundle/...` paths. No other skill has a `bundle/` subdirectory.

- **`consumed-config.md` is a new artifact category.** Previously not tracked anywhere. Each skill now lists, in one structured file: (i) which `research-wiki.config.yaml` keys it reads at every invocation, (ii) keys in its own namespace, (iii) reserved keys not yet used, (iv) what it writes (files, frontmatter edits, appends), (v) what it reads from *other skills' outputs* (cross-skill data-flow visibility), (vi) namespaces it explicitly does *not* consume (boundary verification). The collected form across all 8 skills is the raw material for `docs/CONFIG.md` and `docs/FRONTMATTER.md` (still in "Not yet done").

- **`open-questions.md` now carries explicit `Status:` per entry.** `decided` / `proposed` / `rejected` / `undecided`. Resolved entries stay (for "why" preservation), they don't get deleted. Earlier drafts mixed live and resolved questions in one list, which made the project's deferred-decision surface hard to scan.

- **`P8 enforcement` subsection eliminated.** Previously every `SKILL.md` carried a near-identical "This skill produces analysis, not speculation…" paragraph. The current convention folds that material into (a) the `Principles inheritance` section's P8 bullet and (b) specific behavior-contract bullets. Removing the duplicated subsection saves ~5 lines per skill and removes a maintenance burden (the paragraph subtly drifted across skills).

- **`prompts/writing-skill-md.md` updated** to document the four-file model, the `reference/` conventions, the `bundle/` carve-out, and the revised final-check criteria (which now check `SKILL.md`, `SPEC.md`, and the four `reference/` files separately). The file was renamed in spirit (now covers the whole skill, not just SKILL.md) but its filename stays `writing-skill-md.md` for backward compatibility.

- **Trade-off accepted.** The convention adds files. A skill that used to be 2 files (SPEC + SKILL) is now 6 (SPEC + SKILL + four `reference/` files). The cost is more files to navigate; the benefit is dramatically reduced runtime context per invocation, plus a structured place for material that was previously informal SKILL.md tail content.

- **No outstanding follow-ups from this batch.** Two follow-ups carried from the 2026-04-25 batches remain unaffected: `CLAUDE.md §4` skill count update (deferred to explicit researcher decision), and `wiki-lint` Check #8 orphan refinement honoring `seeded_by:` markers.

### [2026-04-26] `wiki-log` Python + LLM hybrid implementation

This pass closed the only outstanding skill implementation. Seven of the eight skills were already pure-code Python (`src/researchwiki/{init,sync,deepscan,lint,query,recall,fixstale}.py`); `wiki-log` was the lone holdout, deliberately deferred while the question of *how* to implement it was unresolved.

The decision space had three options:

- **(A) Keep `wiki-log` as a pure Claude Code skill (`SKILL.md` only).** Simplest. Cost: deterministic operations (template parsing, frontmatter assembly, atomic writes, code-symbol lookup, log/index/back-ref updates) get re-derived by the LLM each invocation, with all the determinism risks that implies — slug computation drift, frontmatter schema deviation, partial writes on conversation interruption.
- **(B) Push `wiki-log` to pure Python + scripted prompts.** Most consistent with the other seven skills. Cost: the conversational ergonomics (paraphrase italic guides, P8 detection on free Korean prose, identifier extraction from mixed prose) collapse into mechanical heuristics that materially degrade the daily-use feel — and `wiki-log` is the *most-used* skill. Form-filling regression.
- **(C) Python + LLM hybrid.** Mechanical ops in Python, conversation/reasoning in LLM, with the LLM driving the Python toolkit via a JSON-friendly CLI. Highest implementation cost; preserves both ergonomics and determinism.

**Chose (C).** The split is encoded in `ARCHITECTURE.md §3.5` (Class A vs Class B). Concretely:

- **Python core (`src/researchwiki/log.py`, ~600 lines).** Five public functions exposed via the `wiki-log` CLI: `inspect_template` (parse template, compute would-be path, surface workspace state), `lookup_code_symbols` (deterministic match against `index/signatures.json`, supports bare names, dotted names, and bare paths; ambiguous tokens return `matched: false` for researcher disambiguation), `find_pages` (exact-slug lookup against `wiki/<kind>/`), `find_amend_target` (most-recent same-type entry within configurable window), `run_log` (atomic write phase: entry file + log.md append + index.md insertion + bidirectional back-refs + concept stub seeding + questions.md append). Validation enforced in code: `authored_by: llm` is rejected; required sections must have non-empty answers; path collisions raise `FileExistsError`.

- **Template parsing fixed two real-world quirks.** Existing bundle templates (`skills/wiki-init/reference/bundle/templates/<lang>/*.md`) start with an HTML comment block before the YAML frontmatter (researcher-facing notes about how the template is consumed), and the YAML carries unquoted `{{KEY}}` placeholders that PyYAML mis-parses as flow-style mappings. The parser strips leading comments first, then quotes `{{KEY}}` patterns before YAML parse, then runs placeholder substitution downstream when materializing the entry. Body offsets are computed from the un-quoted text so substituted-string lengths don't drift the body slice.

- **LLM side (`skills/wiki-log/SKILL.md` + 9 reference docs).** `SKILL.md` documents the CLI surface and the payload schema, then defers reasoning details to `reference/`. New reference docs (5): `p8-detection.md` (KO/EN hedge-verb / causal-claim / intent-attribution patterns + section-by-section risk map + counter-examples), `conversational-style.md` (italic-guide paraphrasing, pacing, push-back patterns, language register), `auto-link-extraction.md` (identifier-token / experiment-ID / paper-slug / concept-noun-phrase extraction rules + per-section tuning), `refusal-patterns.md` (8 refusal classes with KO/EN templates and alternative-route routing), `templates-deep-dive.md` (per-template strong-vs-weak answer guidance, anti-patterns, when-to-redirect). Existing 4 docs (`examples.md`, `failure-modes.md`, `open-questions.md`, `consumed-config.md`) preserved.

- **CLI dispatch.** Five subcommands wired through `src/researchwiki/cli.py:log_main`. Top-level dispatcher and `pyproject.toml` `[project.scripts]` updated. Pre-existing bug fixed: `src/researchwiki/__main__.py` was calling `main()` without `raise SystemExit(...)`, so `python -m researchwiki <subcommand>` always returned 0 regardless of the subcommand's intended exit code; integration test for collision refusal flushed this out.

- **Tests.** 23 unit tests in `tests/test_log.py` (template parsing including KO frontmatter quirks, all 5 public functions including ambiguous-token / past-window / collision edge cases, payload validation including `authored_by: llm` rejection and blank-required rejection, atomic write side effects, back-ref idempotence, stub-creation skip when target exists). 1 new integration scenario in `tests/integration/test_scenario_wiki_log_full.py` (S5) exercising the full CLI flow end-to-end via subprocess plus a collision-refusal sub-test. Project total: 168 passing (up from 143).

- **Trade-off explicitly noted.** The hybrid is one-skill-deep: only `wiki-log` is Class B. Adding more Class B skills would multiply the LLM-side reasoning surface to maintain (each requiring its own `reference/` library and risking drift between skills). The current bet is that *interpretation of researcher prose* is the only LLM-essential operation in the entire skill set; the seven Class A skills operate on structured artifacts where mechanical pattern-matching is sufficient. This bet is testable: if a future Class A skill (say, `wiki-export`) required free-prose synthesis, it would either be promoted to Class B or refused as out-of-scope per P8.

- **Not done by this batch.** Automated end-to-end testing of the *LLM-side reasoning* (P8 detection accuracy on real prose, conversational fluidity, extraction precision against gold-standard payloads) would require infrastructure that does not exist in the current pure-pytest setup — either real Claude API calls (cost, nondeterminism, API key management), HTTP-cassette replay (recording infrastructure, brittle on model upgrades), or scripted LLM mocks (only validates the LLM↔CLI protocol, not reasoning quality). Filed as deferred for v0.1; manual smoke runs are the current discipline. The reference docs are the authoritative LLM guidance and were written to be read by the LLM at runtime, but their *effectiveness* in producing P8-compliant entries is not empirically validated by the CI.
